**bindings/js/src/loader.ts:**
Implement lazy-loading singleton pattern (per user decision: lazy loading on first use, not on import).

1. Define constants: `CORE_LIB = "libquiver.{suffix}"`, `C_API_LIB = "libquiver_c.{suffix}"` using `suffix` from bun:ffi
2. Implement `getSearchPaths(): string[]` that returns candidate directories:
   - Walk up from `import.meta.dir` (bindings/js/src/) looking for `build/bin/` (development mode -- walk up to 5 levels)
   - Check each candidate with `existsSync(join(candidate, C_API_LIB))`
   - Per user decision: auto-detect from package location first, then system PATH fallback
3. Implement `loadLibrary()` that:
   - Returns `_library` if already cached (lazy singleton)
   - Iterates search paths, for each: check file exists, on Windows pre-load CORE_LIB via `dlopen(corePath, {})`, then `dlopen(cApiPath, symbols)` with all Phase 1 FFI symbol definitions
   - Falls back to bare library name (system PATH)
   - On failure: throws `QuiverError` listing all searched paths and missing library name (per user decision)
4. Export `getSymbols()` convenience function that calls `loadLibrary()` and returns `.symbols`
5. FFI symbol definitions for Phase 1 (lifecycle only -- later phases extend):
   - `quiver_version`: args=[], returns=FFIType.cstring
   - `quiver_get_last_error`: args=[], returns=FFIType.ptr (NOT cstring -- we want the raw pointer to check for null)
   - `quiver_clear_last_error`: args=[], returns=FFIType.void
   - `quiver_database_from_schema`: args=[ptr, ptr, ptr, ptr], returns=FFIType.i32
   - `quiver_database_from_migrations`: args=[ptr, ptr, ptr, ptr], returns=FFIType.i32
   - `quiver_database_close`: args=[ptr], returns=FFIType.i32

**bindings/js/src/errors.ts:**
1. `QuiverError` class extending `Error` with `this.name = "QuiverError"` in constructor
2. `check(err: number): void` function:
   - If `err !== 0`: call `getSymbols().quiver_get_last_error()`, read pointer via `new CString(errPtr)`, throw `QuiverError(message)`
   - If errPtr is null/0: `console.warn(...)` and throw `QuiverError("Unknown error")` (matches Dart pattern)
   - Import `CString` from "bun:ffi" and `getSymbols` from "./loader"

**bindings/js/src/index.ts:**
Barrel export for public API. For Phase 1 export only `QuiverError` from "./errors". The Database class export will be added in Plan 02.

  <name>Task 2: Create FFI helper utilities</name>

  <action>
Create out-parameter allocators, struct builders, and type conversion helpers in ffi-helpers.ts. These are the low-level primitives that Database methods will use.

**Struct builder (FFI-03, FFI-04):**
1. `makeDefaultOptions(): Uint8Array` -- Construct the 8-byte `quiver_database_options_t` struct manually (DO NOT call quiver_database_options_default -- it returns struct by value which bun:ffi cannot handle). Layout: offset 0 = int32 read_only (default 0), offset 4 = int32 console_level (default 1 = QUIVER_LOG_INFO). Use `new ArrayBuffer(8)` + `new Int32Array(buf)`.

**Out-parameter allocators (FFI-03):**
2. `allocPointerOut(): BigInt64Array` -- Allocate a 1-element BigInt64Array (8 bytes) to receive a `quiver_database_t**` out-parameter. Returns the typed array (caller passes `ptr(result)` to FFI).
3. `readPointerOut(buf: BigInt64Array): number` -- Read the pointer value from an out-parameter buffer using `read.ptr(ptr(buf), 0)`. Returns a JS number (pointer). Import `ptr`, `read` from "bun:ffi".

**String conversion helpers (FFI-04):**
4. `toCString(str: string): Buffer` -- Encode a JS string as null-terminated UTF-8 for C API. Returns `Buffer.from(str + "\0")`. This is a thin wrapper but makes intent clear at call sites and avoids forgetting the null terminator.

**Export all functions.** Import `ptr`, `read` from "bun:ffi".

Note: Do NOT create helpers for reading C-allocated arrays (int64_t**, double**, char**) yet -- those are Phase 3 (Read Operations). Phase 1 only needs pointer-to-pointer out-params for database handles.
