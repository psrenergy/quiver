**bindings/js/src/database.ts:**
Implement the Database class following the established pattern from Dart/Python bindings.

1. Private constructor `private constructor(ptr: number)` -- ptr is the opaque native pointer as a JS number (per bun:ffi convention, pointers are numbers, NOT BigInt)
2. Private fields: `_ptr: number`, `_closed: boolean = false` (per user decision)
3. Private method `ensureOpen(): void` -- if `_closed`, throw `new QuiverError("Database is closed")` (exact message per user decision)
4. `static fromSchema(dbPath: string, schemaPath: string): Database`:
   - Call `getSymbols()` to get FFI symbols (triggers lazy library load -- per user decision)
   - Call `makeDefaultOptions()` to get the 8-byte options buffer
   - Call `allocPointerOut()` for the out_db parameter
   - Call the C API: `check(lib.quiver_database_from_schema(toCString(dbPath), toCString(schemaPath), ptr(options), ptr(outDb)))`
   - Read the database pointer: `readPointerOut(outDb)`
   - Return `new Database(dbPtr)`
   - Import `ptr` from "bun:ffi" for passing typed arrays as pointers
5. `static fromMigrations(dbPath: string, migrationsPath: string): Database`:
   - Same pattern as fromSchema but calls `quiver_database_from_migrations`
6. `close(): void`:
   - If `_closed`, return immediately (idempotent per user decision)
   - Call `check(lib.quiver_database_close(this._ptr))`
   - Set `_closed = true`
7. No public getter for `_ptr` (per user decision: native pointer stays private). Add a package-internal accessor pattern: `/** @internal */ get _handle(): number` for use by future Database method files (create.ts, read.ts in later phases). This keeps the pointer off the public API surface while allowing internal module access.
