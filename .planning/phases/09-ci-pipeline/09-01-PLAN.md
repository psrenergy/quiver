The job structure:

```yaml
  deno-test:
    name: Deno Test
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v6

      - uses: denoland/setup-deno@v2
        with:
          deno-version: v2.x

      - name: Build C++ (for shared library)
        run: |
          cmake -B build \
            -DCMAKE_BUILD_TYPE=Debug \
            -DQUIVER_BUILD_TESTS=ON \
            -DQUIVER_BUILD_C_API=ON
          cmake --build build

      - name: Run Deno tests
        working-directory: bindings/js
        env:
          LD_LIBRARY_PATH: ${{ github.workspace }}/build/lib
        run: deno task test
```

Key details:
- `denoland/setup-deno@v2` with `deno-version: v2.x` (current stable Deno 2.x line).
- C++ build step is identical to julia-coverage/dart-coverage/python-coverage (Debug, TESTS=ON, C_API=ON).
- `LD_LIBRARY_PATH` set to `${{ github.workspace }}/build/lib` so `Deno.dlopen` finds `libquiver_c.so` (same env var as julia-coverage job).
- `deno task test` executes `deno test --allow-ffi --allow-read --allow-write --allow-env test/` as defined in `bindings/js/deno.json`.
- Ubuntu-only (multi-platform is deferred to CI-04 in v2 requirements).
- No coverage collection (can be added later, keep this focused on CI-01).

Do NOT add `needs: build` or any dependency on the matrix build job -- the deno-test job builds its own C++ libs locally (same pattern as other binding jobs).

Modify `.github/workflows/publish.yml` to remove the npm publishing pipeline and add a JSR publish job.

**Remove these two jobs entirely:**
1. `build-native-npm` (lines 104-134) -- builds native libs for npm bundling. JSR publishes TypeScript source, not native binaries, so this is unnecessary.
2. `publish-npm` (lines 136-181) -- validates package.json version, downloads native libs, runs npm install/build/publish. All npm references.

**Add this new job after the existing `publish` (PyPI) job:**

```yaml
  publish-jsr:
    name: Publish to JSR
    runs-on: ubuntu-latest
    needs: validate
    environment: jsr
    permissions:
      contents: read
      id-token: write
    defaults:
      run:
        working-directory: bindings/js
    steps:
      - uses: actions/checkout@v6

      - uses: denoland/setup-deno@v2
        with:
          deno-version: v2.x

      - name: Validate tag matches deno.json version
        run: |
          TAG_VERSION="${GITHUB_REF_NAME#v}"
          PKG_VERSION=$(deno eval "const c = JSON.parse(Deno.readTextFileSync('deno.json')); console.log(c.version)")
          if [ "$TAG_VERSION" != "$PKG_VERSION" ]; then
            echo "::error::Tag version ($TAG_VERSION) != deno.json version ($PKG_VERSION)"
            exit 1
          fi
          echo "Version validated: $TAG_VERSION"

      - name: Publish to JSR
        run: npx jsr publish --allow-slow-types
```

Key details:
- `permissions.id-token: write` enables OIDC authentication with JSR (no manual tokens needed).
- `permissions.contents: read` is the minimum for checkout.
- `needs: validate` -- chains after the existing Python validation job so the GitHub Release is created first. The JSR publish is independent of PyPI but sharing the validate gate ensures the tag version matches pyproject.toml too.
- `environment: jsr` -- create a deployment environment for audit trail (GitHub will auto-create it on first use if it does not exist).
- Version validation reads from `deno.json` (not package.json) using `deno eval` to parse JSON.
- `--allow-slow-types` is required per Phase 8 findings (ambient module augmentation pattern).
- No `--allow-dirty` flag -- the tagged commit should have a clean tree.
- JSR publishes TypeScript source files as specified in `deno.json` `publish.include` -- no build step, no native lib bundling.

**Verify after editing:** No occurrences of `package.json`, `npm install`, `npm run build`, `npm publish`, `setup-node`, `build-native-npm`, or `publish-npm` remain anywhere in publish.yml.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions OIDC -> JSR | OIDC token from GitHub is exchanged with JSR for publish authorization |
| GitHub Actions OIDC -> PyPI | Existing boundary (unchanged) |
| Workflow trigger -> job execution | Push/PR triggers and tag triggers control which jobs run |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Spoofing | OIDC token for JSR publish | mitigate | `id-token: write` permission scoped to `publish-jsr` job only (not workflow-level). `environment: jsr` adds deployment protection rules. Tag trigger `v*.*.*` limits when publish runs. |
| T-09-02 | Elevation of Privilege | Deno test permissions | mitigate | `deno task test` uses explicit permission flags (--allow-ffi --allow-read --allow-write --allow-env) defined in deno.json. No --allow-all. Test job has no write permissions to anything outside the workspace. |
| T-09-03 | Tampering | Supply chain via denoland/setup-deno action | accept | Using `@v2` tag (major version pin). Same trust model as existing `julia-actions/setup-julia@v2`, `dart-lang/setup-dart@v1`, `actions/setup-node@v6`. Risk is accepted as standard for GitHub Actions. |
| T-09-04 | Information Disclosure | LD_LIBRARY_PATH exposure | accept | LD_LIBRARY_PATH points to workspace build directory only. No secrets exposed. Standard pattern used by julia-coverage job already. |
| T-09-05 | Denial of Service | CI resource consumption | accept | Deno test job runs on ubuntu-latest only (not matrix). Build is Debug mode (same as other binding jobs). Bounded CI time. |
</threat_model>
