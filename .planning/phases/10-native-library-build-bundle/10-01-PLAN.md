<!-- Key runtime behavior the executor must understand -->

From bindings/js/src/loader.ts (lines 174-181):
```typescript
function getBundledLibDir(): string | null {
  const platformKey = `${Deno.build.os}-${Deno.build.arch}`;
  const libDir = join(__dirname, "..", "libs", platformKey);
  if (fileExists(join(libDir, C_API_LIB))) {
    return libDir;
  }
  return null;
}
```

CRITICAL: Deno.build.os returns "linux" or "windows" (not "win32").
CRITICAL: Deno.build.arch returns "x86_64" (not "x64").
Therefore platform keys at runtime are: linux-x86_64, windows-x86_64.
The CI artifact directories MUST use these exact names.

From CMakeLists.txt: Output directories by platform:
- Linux: CMAKE_LIBRARY_OUTPUT_DIRECTORY = build/lib (libquiver.so, libquiver_c.so)
- Windows: CMAKE_RUNTIME_OUTPUT_DIRECTORY = build/bin (DLLs land in build/bin/Release/ with MSVC multi-config)

From cmake/Platform.cmake: Linux .so files need RPATH patched via patchelf so libquiver_c.so finds libquiver.so in the same directory at load time.

Add a new `build-native` job to `.github/workflows/publish.yml` that builds the C++ libraries for Linux x86_64 and Windows x86_64 in Release mode.

The job uses a matrix strategy with two entries:
  - os: ubuntu-latest, platform_key: linux-x86_64, lib_glob: build/lib/libquiver*
  - os: windows-latest, platform_key: windows-x86_64, lib_glob: build/bin/Release/libquiver*

Steps for the build-native job:
1. actions/checkout@v6
2. Configure CMake: `cmake -B build -DCMAKE_BUILD_TYPE=Release -DQUIVER_BUILD_C_API=ON -DQUIVER_BUILD_TESTS=OFF`
   - On Windows, use backtick line continuation (PowerShell default shell on windows-latest).
   - On Linux, use backslash line continuation.
   - Use conditional `if: runner.os == 'Linux'` / `if: runner.os == 'Windows'` steps (same pattern as ci.yml).
3. Build: `cmake --build build --config Release`
4. (Linux only) Patch RPATH: `patchelf --set-rpath '$ORIGIN' build/lib/libquiver_c.so` so libquiver_c.so finds libquiver.so when both are in the same directory. Install patchelf first: `sudo apt-get update && sudo apt-get install -y patchelf`.
5. Upload artifact: actions/upload-artifact@v7 with name `native-${{ matrix.platform_key }}` and path `${{ matrix.lib_glob }}`.
   On Linux this uploads build/lib/libquiver.so and build/lib/libquiver_c.so.
   On Windows this uploads build/bin/Release/libquiver.dll and build/bin/Release/libquiver_c.dll.

The job has no `needs:` (runs in parallel with build/merge/validate).

Per CI-05: This job runs before publish-jsr (enforced in Task 2 via needs).
Per CI-06: CMake uses Release mode with QUIVER_BUILD_C_API=ON.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI build -> JSR package | Native binaries built in CI are bundled into the published package |
| JSR package -> consumer runtime | Consumer imports package and loads bundled native .so/.dll files |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | CI build artifacts | mitigate | GitHub Actions artifact upload/download uses authenticated API within the same workflow run; artifacts cannot be replaced by external actors. Build uses pinned action versions (@v6, @v7, @v8). |
| T-10-02 | Tampering | Native binaries in transit | accept | JSR publishing happens in the same workflow run that built the binaries -- no external artifact store. OIDC auth prevents unauthorized publish. Binaries are never persisted to git. |
| T-10-03 | Elevation of Privilege | Malicious native library | mitigate | Only CI-built binaries from the repo's own CMake build are bundled. No external binary downloads. The build-native job checks out the same tagged commit. publish-jsr uses `environment: jsr` requiring environment approval. |
| T-10-04 | Information Disclosure | Build artifacts contain debug symbols | mitigate | CMake uses Release mode (-DCMAKE_BUILD_TYPE=Release) which strips debug info by default. No PII in binaries. |
| T-10-05 | Denial of Service | Large binary bloats package | accept | Two platforms x 2 shared libraries is ~4 files. Acceptable package size for native FFI library. macOS/ARM64 deferred. |
</threat_model>
